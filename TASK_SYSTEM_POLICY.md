# Task 시스템 배정 정책 문서

> 이 문서는 트랙 생성 → Task 자동 생성 → 학관매 배정 → 수행 → 팔로업까지의 전체 파이프라인에 대한 정책을 정리한 것입니다.
> 새 에이전트나 개발자가 이 문서만 읽으면 Task 시스템의 모든 맥락을 파악할 수 있도록 작성되었습니다.

---

## 1. 프로젝트 목적과 Task 시스템의 역할

### 프로젝트 목적
1. **학관매(학습관리매니저)** 자리에 누가 와도 퀄리티를 유지할 수 있도록, 미리 정의된 Task를 자동 배정하고 팔로업 가능한 구조를 만든다
2. **총괄(운영기획매니저)** 의 리소스를 대폭 절감하여, 기본적인 운영이 잘 돌아가는지 지표만 보고도 확신할 수 있게 한다

### Task 시스템이 하는 일
- 트랙 생성 시 **63개 사전 정의 템플릿**을 기반으로 Task 인스턴스를 자동 생성
- 빈도(매일/주간/챕터별/월별/1회성/수시)에 따라 인스턴스 수가 결정됨
- 생성된 Task를 학관매에게 배정 (전원/지정/순환/미지정 모드)
- 학관매가 매일 대시보드에서 "오늘 할 일"을 확인하고 수행
- 운영매니저가 완료율/이슈를 모니터링하고, 미처리 시 에스컬레이션

## 2. 역할 정의

| 역할 키 | 풀네임 | 줄임말 | 설명 |
|---------|--------|--------|------|
| `operator_manager` | 운영기획매니저 | 총괄 | 4~5개 트랙을 총괄. 운영매니저만 관리 |
| `operator` | 운영매니저 | 운영 | 학관을 직접 관리. 트랙 운영 실무 |
| `learning_manager` | 학습관리매니저 | 학관 | 수강생과 직접 소통하며 일일 Task 수행 |

### 정책 결정 사항
- **H.학관매(헤드 학관매)**: 별도 역할이 아님. CSV 데이터에서 "H.학관매"로 표기된 것은 모두 **운영매니저(`operator`)**와 동일
- **튜터**: MVP에서 제외. CSV에서 DRI가 "튜터"인 Task는 `learning_manager`로 매핑
- DriRole 타입: `'operator_manager' | 'operator' | 'learning_manager'` (tutor 없음)

### 조직 구조
```
총괄 1명
 └─ 운영매니저 3명 (각각 1~2개 트랙 담당)
     └─ 학관매 3명/트랙 (각 운영당 3명)
```

## 3. Task 템플릿 구조

### 데이터 소스
- 파일: `lib/task-templates.ts` (63개 템플릿)

- 게임 트랙 전용 (#64, #65)은 제외

### TaskTemplateConfig 타입

```typescript
interface TaskTemplateConfig {
  id: string
  number: number
  category: string
  subcategory: string
  title: string
  description?: string
  driRole: DriRole
  subDriRole?: DriRole
  frequency: TaskFrequency
  triggerType: string
  triggerOffset?: number
  priority: TaskPriority
  scope?: TaskScope
  reviewer?: string
  estimatedDays?: number
  output?: string

  // CSV에서 추가될 필드 (향후 확장)
  detailedDescription?: string   // "학관매 업무 상세" — 수행 방법 안내
  specialNotes?: string          // "특이사항" — 예외/주의사항
  input?: string                 // "인풋" — 시작 전 필요한 전제조건
  triggerDescription?: string    // "업무 트리거" 텍스트 ("개강 D-7" 등)
  deadline?: string              // "종료일 (MUST)" 텍스트
}
```

### 빈도 유형 (TaskFrequency)

## 5. Task 배정 정책 (핵심)

### 배정 모드 (4가지)

트랙 생성 시, 각 Task 템플릿마다 배정 모드를 설정한다.

| 모드 | 코드값 | 설명 | 인스턴스 생성 | 예시 |
|------|--------|------|-------------|------|
| **전원** | `all` | 모든 학관매가 각자 수행 | 학관매 수 × 인스턴스 수 | 출결 확인, 순회 |
| **지정** | `specific` | 특정 학관매 1명에게 고정 | 1 × 인스턴스 수 | 밍글데이 준비 |
| **순환** | `rotation` | 학관매끼리 번갈아 수행 | 1 × 인스턴스 수 (배정자만 다름) | TIL 선정, VOC |
| **미지정** | `unassigned` | 운영 중 운영매니저가 수동 배정 | 1 × 인스턴스 수 (assignee 없음) | 이슈 대응 |

### "전원" 모드 상세

- 학관매가 3명이면, daily Task 1개 템플릿 → 매일 3개 인스턴스 생성
- 각 인스턴스에 `assigneeId`가 각각 학관매 ID로 지정됨
- 예: "출결 확인" → 김학관(staff1), 이학관(staff2), 박학관(staff3) 각각 1개씩

### "순환" 모드 상세

```typescript
interface RotationConfig {
  cycle: 'daily' | 'weekly'     // 순환 주기
  staffOrder: string[]           // 학관매 ID 순서 ["staff1", "staff2", "staff3"]
}
```

- `daily` 순환: 월=A, 화=B, 수=C, 목=A, 금=B, (다음주) 월=C ...
- `weekly` 순환: 1주차=A, 2주차=B, 3주차=C, 4주차=A ...
- 인스턴스는 1세트만 생성되고, `assigneeId`가 순환 규칙에 따라 결정됨

### "지정" 모드 상세

- 트랙 생성 시 특정 학관매 1명을 선택
- 해당 학관매의 ID가 모든 인스턴스의 `assigneeId`에 들어감
- 학관매 휴가 시: `unassigned`로 전환, 운영매니저가 수동 재배정

### "미지정" 모드 상세

- `assigneeId`가 비어있는 상태로 생성
- 트랙 상세 페이지의 "미배정 Task" 영역에 표시
- 운영매니저가 운영 중 필요할 때 수동 배정
- 미배정 상태 3일 이상 지속 시 운영매니저에게 알림 (SYS-02)

## 6. Task 배정 위저드 단계

### 위저드 플로우 (변경 후)

```
Step 1: 기본 정보 (트랙명, 회차)
Step 2: 시간표 업로드 (xlsx 파싱)
Step 3: 챕터 편성
Step 4: 인원 배정 (학관매 이름 등록)
Step 5: Task 생성 (매일/주간/챕터별/월별/일정 아코디언)
Step 6: Task 배정 ← [신규] 학관매별 배정 모드 + 담당자 설정
Step 7: Task 확인 (읽기 전용 요약)
Step 8: 최종 확인
```

### 기본값 (Best Practice)

모든 Task를 처음부터 "미지정"으로 시작하면 설정이 너무 번거로움.
시스템이 Task 특성에 따라 합리적인 기본값을 제안한다:

| Task 특성 | 기본 배정 모드 | 이유 |
|-----------|---------------|------|
| daily + 학관매 DRI + 우선순위 상/중 | **전원** | 출결, 순회 등 전원이 매일 해야 하는 것 |
| daily + 학관매 DRI + 우선순위 하 | **순환 (일별)** | TIL, VOC 등 한 명씩 돌아가며 해도 되는 것 |
| weekly + 학관매 DRI | **순환 (주별)** | 주간 업무는 번갈아 담당 |
| per_chapter + 학관매 DRI | **지정** | 챕터별 리포트 등 연속성 필요 |
| once + 운영매니저 DRI | **미지정** | 운영매니저가 직접 수행 |
| ad_hoc | **미지정** | 발생 시 배정 |

## 7. 수시 Task 정책

### 정의
- `frequency: 'ad_hoc'`인 15개 템플릿
- 정해진 날짜 없이 "필요할 때" 수행하는 업무
- 예: 이탈 위험군 판단, 팀 갈등 중재, 중도하차자 면담

### 트랙 생성 시 처리

1. Step 5 "트랙 일정 Task" 섹션 끝에 **"수시 업무 가이드"** 패널을 표시
2. 각 수시 Task에 대해 선택지 제공:
   - **"날짜 지정"**: 특정 날짜에 1회성으로 생성
   - **"반복 설정"**: daily/weekly 등으로 변환하여 반복 생성
   - **"건너뛰기"**: 생성하지 않음 (트랙 운영 중 필요 시 수동 추가)
3. 건너뛴 수시 Task는 **생성되지 않음** (placeholder 없음)

### 가이드 문구

```
수시 업무는 트랙 운영 중 필요할 때 수행하는 업무입니다.
지금 날짜를 지정하지 않아도, 트랙 운영 중 언제든지 추가할 수 있습니다.
```

## 8. Task 상태 전환

```
[생성] → unassigned → pending → in-progress → completed
                                      ↓
                                   overdue → completed
```

| 전환 | 트리거 | 알림 |
|------|--------|------|
| 생성 → unassigned | 트랙 생성, assignee 없음 | — |
| 생성 → pending | 트랙 생성, assignee 있음 | ACT-06 (학관에게) |
| unassigned → pending | 운영매니저가 배정 | ACT-06 |
| pending → in-progress | scheduledDate 당일 도래 (자동) | SYS-03 |
| in-progress → completed | 학관매가 완료 체크 | ACT-05 (운영에게) |
| in-progress → overdue | dueDate 경과 (자동) | SYS-01 (학관+운영) |
| overdue → completed | 뒤늦은 완료 | ACT-05 |

### 정책 결정 사항
- **완료 확인**: 학관매 자체 완료 처리 (운영매니저 승인 불필요)
- **운영매니저 역할**: 완료율 지표로 모니터링, 이상 시 개입
- **Task 시간 변경 권한**: 학관매는 본인 Task의 완료만 가능. 스케줄(시간/날짜) 변경은 운영매니저만 가능

---

## 9. 업무완료율 계산

### 공식

```
업무완료율 = (오늘 completed Task 수) / (오늘 scheduledDate인 전체 Task 수) × 100
```

- **분모**: 오늘 `scheduledDate`인 Task 중 해당 인원에게 배정된 것 전부
- **분자**: 그 중 `status === 'completed'`인 것
- **"전원" 모드 Task**: 각 학관매별 인스턴스가 따로 있으므로 자연스럽게 개인별 계산

### 임계치 알림

- **학관매 업무완료율** (THR-01): 오후 2시 기준, 30% 미만 → 담당 운영매니저에게 알림
- **운영매니저 업무완료율** (THR-02): 오후 2시 기준, 30% 미만 → 총괄에게 알림
- 임계치와 체크 시점은 `/admin/notifications` 알림 설정에서 트랙별 조절 가능

## 10. 학관매 휴가 처리

1. 운영매니저가 학관매 휴가를 등록 (ACT-13 알림 발생)
2. 해당 학관매의 휴가 기간 Task → `unassigned`로 전환
3. 운영매니저가 다른 학관매에게 **수동 재배정**
4. 휴가 복귀 시 운영매니저가 수동으로 다시 배정
5. "전원" 모드 Task는 자동으로 해당 학관매의 인스턴스만 unassigned 처리

## 13. 스토어 구조 (SSOT 통합 방향)

### 현재 문제
3개의 독립 스토어가 데이터를 공유하지 않음:

| 스토어 | 사용처 | 데이터 소스 |
|--------|--------|------------|
| `admin-store.ts` | 총괄/운영 대시보드, 트랙 상세 | `admin-mock-data.ts` |
| `store.ts` | 학관 대시보드 | `mock-data.ts` |
| `interview-store.ts` | 면담 탭 | `interview-mock-data.ts` |

### 결과
- 위저드에서 Task를 만들어도 학관 대시보드에 안 나타남
- 운영매니저가 이슈에 답변해도 학관 화면에 반영 안 됨
- 면담 데이터가 트랙/학관과 무관

### 통합 방향
- `admin-store.ts`를 **Single Source of Truth**로 사용
- `store.ts`는 제거하고, 학관 대시보드는 admin-store에서 `staffId`로 필터링
- `interview-store.ts`에 `trackId`/`staffId` 연결 추가

```typescript
// 학관 대시보드용 파생 데이터 훅
function useStaffData(staffId: string) {
  const trackTasks = useAdminStore(s => s.trackTasks)
  const today = format(new Date(), 'yyyy-MM-dd')

  const myTasks = trackTasks.filter(t => t.assigneeId === staffId)
  const todayTasks = myTasks.filter(t => t.scheduledDate === today)
  const timedTasks = todayTasks.filter(t => t.scheduledTime)
  const untimedTasks = todayTasks.filter(t => !t.scheduledTime)

  return { myTasks, todayTasks, timedTasks, untimedTasks }
}
```

### 라우트 파라미터 활용
현재 `/staff/[id]`, `/managers/[id]`, `/operators/[id]`에서 `params.id`가 무시되고 있음.
통합 후 `params.id`를 사용하여 해당 인원의 데이터만 필터링.
